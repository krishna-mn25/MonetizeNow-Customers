Self-Service

MonetizeNow's self-service capabilities let you build product-led checkout flows, trial experiences, and billing portals directly on top of the API — without routing customers through a sales motion.

What you can build

Checkout flows
Let customers sign up, pick a plan, and start a subscription without a rep involved.
Trials
Time-bound free access that converts to a paid subscription automatically or on demand.
Billing portals
Customer-facing portals to manage invoices, payment methods, and subscriptions.

How it works

Every self-service flow runs through MonetizeNow's API. Your frontend or backend calls the API to create accounts, spin up quotes or trials, collect payment, and trigger subscription creation. MonetizeNow handles billing logic, contract lifecycle, and revenue recognition behind it.

Stripe handles payment collection. When a customer enters card details on your checkout, that payment method is attached to their account in MonetizeNow and charges fire automatically at billing dates.

Key concepts

ConceptWhat it is
AccountRepresents a customer. Holds contacts, payment methods, contracts, and trial records.
QuoteDefines what a customer is buying: offerings, pricing, contract length, and terms. A processed quote creates a contract and subscription.
TrialA time-limited free period that carries conversion config so it can become a paid subscription without extra input at conversion time.
ContractCreated when a quote processes. Governs billing for the subscription period.
SubscriptionActive billing relationship created from a contract. Drives invoice generation and payment collection.

Architectural Requirements

Before building a self-service flow on MonetizeNow, a few things need to be in place. Most are one-time setup items.

Prerequisites

Product catalog

At minimum, you need at least one active Offering with an active Rate. Rates define pricing — a Rate at $0 works for free trials. Products define what customers get access to.

Stripe connection

Self-service checkout requires Stripe connected and configured in MonetizeNow settings. Without it, the payment step in checkout cannot complete.

API credentials

All self-service flows call the MonetizeNow API on behalf of a tenant. You need a valid API key scoped to your tenant, generated from API Keys in settings.

What your code is responsible for

  • Creating and identifying customers as MonetizeNow accounts
  • Presenting pricing and collecting plan selection
  • Calling the MonetizeNow API to create quotes, trials, and process subscriptions
  • Collecting payment method via Stripe's SDK, then passing the payment method ID to MonetizeNow
  • Handling webhook events to keep your own system state in sync
MonetizeNow does not host a checkout UI. It provides the billing and contract infrastructure. Your frontend owns the customer-facing experience.

Trials

Trials give customers a free evaluation window before billing starts. MonetizeNow manages the full lifecycle: creation, extension, conversion to a paid subscription, and cancellation.

What is a trial?

A trial is a standalone record in MonetizeNow, separate from quotes and subscriptions. It carries everything needed to become a paid subscription, but nothing about billing runs until conversion. Trials are always free. Billing starts from the contract start date on the subscription created at conversion.

When you create a trial, you configure two things up front: what the paid quote will look like, and which products to convert onto. Both are locked at creation. At conversion time, MonetizeNow replays that configuration automatically.

Creating a trial

Trials are created via API. Everything needed to run the trial and convert it to a subscription is configured at creation — nothing needs to be provided again at conversion time.

What you configure at creation

POST /trials your API call
▶

▶
at conversion
Replayed automatically builds quote + offerings processes subscription no extra input needed

Quote initialization details

Defines what the paid subscription quote will look like at conversion: contract length, net terms, primary contact, and billing contact.

Conversion policy

Defines the products that go on the converted quote: which offerings, which rate per offering, and product IDs with quantities. All locked at creation and replayed verbatim at conversion.

Required fields

FieldDescription
accountIdAccount this trial belongs to.
typeNEW for a net-new trial.
startDateWhen the trial period begins.
endDateWhen the trial period ends. Can be extended while active.
endActionCONVERT_TO_PAID or CANCEL. Defaults to CANCEL.

Validation at creation

All configuration is validated before the trial is written. If anything fails, nothing is created.

  • All offerings must be active and within their active date range
  • All rates must be active and within their active date range
  • Account-scoped rates must belong to the same account as the trial
  • All required products on each rate must be included
  • Product quantities must satisfy min and max rules on each rate
  • One-time offering types are not permitted

Converting to paid

When a trial converts, MonetizeNow builds and processes a full paid quote automatically using configuration stored at trial creation. No additional input is required.

Quote stages: pending → review → approve → sent → accepted → processed. All run automatically. No manual approvals needed.
1
Validate the trial is active
Conversion is blocked if the trial is in any terminal status. Only ACTIVE trials proceed. This guards against race conditions between batch and manual calls.
2
Build the quote from quoteInitDetails
A net-new quote is created. Contract start is set to the current date. Contract length, net terms, and contacts come from the trial's stored quoteInitDetails.
3
Add offerings from the conversion policy
Each entry in trialConversionPolicy becomes a quote offering: offering ID, rate, products, and quantities applied exactly as stored at trial creation.
4
Auto-process through all quote stages
MonetizeNow advances the quote automatically through pending, review, approve, sent, accepted, and processed. No manual approvals are needed.
5
Update the trial record
Trial status becomes AUTO_CONVERTED or MANUALLY_CONVERTED. Resulting quote ID is written to the trial, linking it permanently to its subscription.

When billing starts

No charges apply during the trial period. Billing starts from the contract start date on the subscription created at conversion, which is set to the date conversion runs.

Current limitation: converted quotes are always created in USD. Multi-currency trial conversion is not yet supported.

Extending a trial

Active trials can have their end date pushed forward. Useful when a customer needs more evaluation time before committing.

How it works

Call the extend endpoint with a new end date. MonetizeNow replaces the current end date and the trial continues as normal. Two rules apply:

  • Only ACTIVE trials can be extended. Terminal status trials cannot be changed.
  • New end date must be after the current end date. Extension cannot shorten a trial.
Extending a trial does not change its endAction. A trial set to CONVERT_TO_PAID will still auto-convert at the new end date.

Multiple extensions

Trials can be extended more than once. Each call replaces the current end date. There is no limit on extensions as long as the trial stays active.

Canceling a trial

Trials can be cancelled at any point while active. Cancellation ends the evaluation period immediately without creating a subscription.

How it works

Call the cancel endpoint with a cancellation reason. MonetizeNow marks the trial as MANUALLY_CANCELED and stores the reason and timestamp. Only ACTIVE trials can be cancelled.

  • Billing or credit state
  • Ability to create a new trial for the same account in future

FAQs

Common questions about how trials work in MonetizeNow.

Can I change what products a trial converts onto after it's created?

No. Offering and product configuration is locked at creation. To change what a customer converts onto, cancel the existing trial and create a new one with updated configuration.

What happens if conversion fails?

If conversion fails mid-run, the trial stays ACTIVE and nothing partial is written. Resolve the issue — check offerings and rates are still active, contacts are valid — then retry conversion.

Does converting bypass the normal approval workflow?

Yes. Conversion auto-advances the quote through all stages including approval. If your business requires a manual approval step before a trial converts, manage that before calling the convert endpoint.

Can a trial be extended multiple times?

Yes. Extension can be called as many times as needed while the trial is active. Each call replaces the previous end date.

Can I extend a trial that has already expired or been cancelled?

No. Extension only works on ACTIVE trials. Once a trial reaches any terminal status it can't be changed.

What currency are converted quotes created in?

Converted quotes are always created in USD. Multi-currency trial conversion is a current limitation.

Is there a way to set auto-renewal on the converted subscription?

Not through trial configuration. Converted quotes always have auto-renewal off. It can be configured on the resulting contract after conversion if needed.
