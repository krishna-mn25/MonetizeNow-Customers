Prepaid credit pool
How included spend is modeled in MonetizeNow
1
Customer commits to a prepaid pool, e.g. $500 of included credits per period.
This sits on a contract as a one-time product, not a recurring subscription line.
2
Pool scope is set to usage charges only. It draws down against metered consumption, not flat fees or platform charges.
Credit application runs automatically at invoice time.
3
At invoice time, pool offsets usage charge first. Remaining balance becomes overage.
MN tracks amountCredited and net overage on every invoice line item.

1
In MN product catalog, create a new product and select type: Prepaid Credit.
2
Set credit scope to "Usage products only." This prevents pool from offsetting platform or flat fees.
3
Set credit value to match committed tier (e.g. $500). This becomes pool ceiling.
4
Add this product to a customer quote as a one-time line item alongside a recurring usage product.
Usage / overage product
How metered consumption maps to billing in MonetizeNow
1
Usage events are posted to MN via API throughout each month as credits are consumed.
Each event carries a quantity and a metering dimension key.
2
At end of billing period, MN calculates total consumption and applies your per-unit rate.
Tiered, flat, and volume pricing models are all supported.
3
Usage invoice is generated in arrears, after a period closes.
Controlled via subscriptionTiming = ARREARS on rate configuration.

Card on file · Stripe integration
How auto-charge on overage works in MonetizeNow
1
Customer completes a Stripe setup flow to store a card on file. No charge happens at this stage.
MN creates a Stripe customer and attaches a validated payment method.
2
When a usage invoice finalizes at month end, MN fires an auto-charge via Stripe.
Retry and dunning logic handles failed payments automatically.
3
Failed payments surface in a dunning queue for manual escalation if retries exhaust.

Authorized spend limit
What MN handles vs what lives upstream
MN does not natively block usage mid-period when a spend cap is hit. Authorized limit enforcement lives upstream, in your usage ingestion service, before events are posted to billing.
1
Your system tracks real-time credit consumption against each customer's authorized monthly limit.
2
When approaching a limit (e.g. 90%), your system can alert or throttle before posting more usage events.
3
Once limit is hit, stop posting usage events to MN. MN bills only what it receives.
Responsibility split
Spend cap enforcementYour ingestion layer
Usage event ingestionMonetizeNow API
Subscription suspensionMN billingStatus field
Limit reset (monthly)Your system triggers
Alerts at 80% / 90%Your notification layer
1
Store authorized spend limit per customer in your system. Compare against cumulative usage before each API post.
2
Build threshold alerts at configurable percentages so customers are warned before hitting a cap.
3
To reflect suspension in MN, set billingStatus = SUSPENDED on subscription via API when cap is hit.
4
At start of each new billing period, reset your counter and re-activate subscription (billingStatus = ACTIVE).
Usage metering in MonetizeNow
How consumption is tracked and accumulated during each period
1
Your system posts usage events to MN via API. Each event includes quantity, timestamp, and a dimension key.
2
MN accumulates events against each subscription's usage product throughout a billing period.
3
At billing close, MN resolves pricing against accumulated total. Tiered rates apply automatically.

Instrument your application to emit a usage event whenever credits are consumed.

4
MN handles deduplication and accumulation. No need to aggregate before posting.

Credit application at invoice time
How MN applies prepaid pool against usage
1
When invoice is generated, MN's credit policy applicator runs automatically.
2
Prepaid pool balance is checked. Credits apply to usage charge up to pool value.
3
Remaining balance after credit application is overage. That amount gets charged via Stripe.

3

Overage charge via Stripe
Auto-collect net amount at close
1
Once invoice finalizes, MN triggers a Stripe charge for net overage amount.
2
Charge fires against stored payment method on customer's BillGroup.
3
Successful charge marks invoice paid. Failed charge enters dunning retry cycle.

3
